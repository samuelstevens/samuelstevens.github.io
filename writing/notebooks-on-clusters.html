Python Notebooks on Compute Clusters

I use notebooks (Jupyterlab and Marimo) on a compute cluster (OSC) without a desktop environment. This post describes how I do so.

TL;DR:

  1. Use uv to manage Python environments.
  2. Use tmux to manage shell sessions on the cluster’s login nodes.
  3. For visualizations and light, CPU-only analysis: run notebook kernels in my tmux session on the login nodes.
  4. Use ssh port forwarding to forward the kernel port (typically 8888 for Jupyter and 2718 for Marimo) from the login node to my laptop
  5. Open http://localhost:8888 on my laptop browser.

More details below.

Tmux

Tmux is a terminal multiplexer which means it gives you shell-based tools for managing shell sessions. While much of its functionality can be satisfied by your terminal emulator’s tabs/windows/session management, tmux is really useful for remote session management.

I’m not going to put a whole tmux tutorial here; but this guide is okay. Zellij aims to be a terminal multiplexer with some additional features. Might be worth looking into.

You really need tmux, screen or zellij to keep the notebook kernel running on the remote machine when your ssh connection drops (laptop falls asleep, you move between wifi hotspots, etc.).

Login Nodes vs GPU Nodes

Many Slurm-based clusters separate login nodes (shared in parallel between users, typically large CPU machines) and GPU nodes (typically not shared in parallel between users, typically both strong CPU and GPU hardware). For instance, OSC has two login nodes for the Ascend cluster, each of which have 2x AMD EPYC 7643 48-Core and ~1TB of RAM. The compute nodes have AMD EPYC 7H12 CPUs and NVIDIA A100 40GB GPUs.

For notebooks that are doing simple work (visualizing images, matplotlib plotting, small linear regressions, etc), it’s totally fine to run this on the login node.

Notebooks that are doing GPU work (obviously) have to run on GPU compute nodes via Slurm, and if you plan on doing heavy CPU work, you should submit to a CPU-only compute node because all users shared the same CPUs on the login nodes; if you do some heavy work on the login node, everyone’s editors, shells, etc all slow down.

For this reason, I typically only use notebooks for lightweight work and instead use Python scripts in combination with submitit for submitting Slurm jobs.

There is two-hop port-forwarding, but I won’t cover it because I don’t use notebooks for GPU work.

SSH Port Forwarding

Once the notebook kernel is running on the login node, I will set up SSH port forwarding so that I can use my laptop’s web browser for the notebook interface.

When your browser makes an HTTP request to http://localhost:8888, we want the request to be delivered to http://localhost:8888 on the login node. Since we connected to the login node with SSH, we can use SSH port forwarding to achieve this goal.

It’s pretty simple:

ssh $HOST -N -L $PORT:localhost:$PORT

List of Concrete Actions

  1. In terminal window #1, ssh samuelstevens@ascend-login01.hpc.osc.edu: SSH into login node 01.
  2. In terminal window #1, tmux a -t sae: attach to my existing tmux session with the name ‘sae’.
  3. In terminal window #1, uv run jupyterlab or uv run marimo edit inside the tmux session to start the kernel on the login node.
  4. In terminal window #2, ssh samuelstevens@ascend-login01.hpc.osc.edu -N -L 8888:localhost:8888: set up port forwarding for port 8888.
  5. In a browser on my laptop, open http://localhost:8888.

Notes:

  • I don’t use VS Code to run notebooks; I prefer the native web interface.
  • I don’t run heavy processing inside notebooks; I prefer to write Python scripts and submit them with submitit to Slurm compute nodes.
